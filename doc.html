IA Bet Onboarding IA · Context & Problem

Onboarding IA

Context

Claude.ai is the first entry point of the user into Anthropic's product suite. Successful onboarding and customizing the product is important to retain users and increase the product usage.

Problem

The current onboarding experience

  • Feels fragmented as it takes the user from a centralized list into three completely different product sections
    • Deeply nested Settings modal (Settings > Capabilities > Memory)
    • Separate directory modal under "Customize"
    • A standalone Get Apps and Extensions landing page from the home screen
  • Tasks are marked as completed when users merely open the destination page, even if they don't finish the setup resulting in inflated activation rates
  • Checklist disappears when all tasks are marked "complete". Users don't have a clear path to complete unfinished onboarding steps later

Hypothesis

I believe that IA improvements of the onboarding experience will increase the true activation rate because users can more easily discover and complete the required onboarding steps.

Assumptions

  • Activation rate is when users complete all steps in onboarding
  • Aha moment is when users engage at least in 5 chats to understand the value and uniqueness of claude.ai

Metrics

% True activation rate
# WAU
Counter metric: # Time to achieve aha moment

Scope

3 days of effort · majority backend · 4-week test

Small / Medium Bet Usage Limit Awareness · Problem

Usage Limit Awareness and Best Practices

Problem / Opportunity

Based on my online research (reddit and blogs), Supra community interviews, and my experience, the most common pain point is that new users, especially coming from other AI products, hit the usage limit faster.

When users connect their everyday tools and configure Claude.ai to get better answers, they consume more tokens and hit their limits quickly. Though Claude provided usage limit best practices and controls to optimize model usage (effort level), the lack of awareness of the usage during the chat leads to abrupt interruptions to users with "Claude hit the maximum length for this conversation" error.

Currently, users are checking their usage repeatedly through settings or hacking their way by starting their limit window early. Lack of awareness of the usage is also causing the friction to adapt the best practices effectively.

Most Impacted Users

The most impacted users are the mid journey users who moved past the initial exploration, connected their tools and made Claude.ai their default chat for their everyday research & synthesis and brainstorming but not yet formed the habit to manage limits or upgrade to higher paid plans. Interruption and lock out of their work has a high chance of churn.

Why Now?

As more users are migrating from competitor products and Claude.ai usage is moving towards broader adoption, the usage limit will only become more severe due to compute capacity crunch. While we are solving the compute capacity with partnerships, it is limited to claude code and enterprise users. The majority of users are casual users and will churn if they face the usage limits frequently.

Hypothesis

I believe that awareness of usage level in the chat and progressive nudges will reduce the odds of users getting interrupted, because they are now more aware of their consumption and adjust their usage behavior accordingly. As users continue using the product more uninterruptedly, the retention rate will increase

Metrics

# Users hitting the usage limits
# Average session length per user
# D14 retention rate
# WAU
Counter metric: % of free to paid conversion

Kill switch: If the free to paid conversion drop is significant among test cohort

Small / Medium Bet Usage Limit Awareness · Solution

Proposed Solution

Proposed Solution

Proposed solution solves the pain points of

  • Awareness of usage
  • Inform the best practices of context management so users can get better answers from Claude without hitting the limits

User Experience

Show the persistent usage meter in the chat interface for all plans similar to claude code desktop sessions.

When 75% of the length limit of the chat is reached, inform the user about the approaching limit and show below progressive nudges

Progressive Nudges — Priority Order

NudgeUser BenefitBusiness Benefit
If a user hasn't enabled the code execution settings, nudge the user to enable the settings. Enables auto summary so chat context is not lost and keeps the continuity. It also unlocks the creation of artifacts Deepens the usage of Claude as more use cases (such as file creation/analysis) are unlocked.
If a user already enabled the code execution setting and chat is not in project, nudge the user to create a project to manage this and subsequent chats as part of best practice Can work with large set of information effectively without hitting the limits Adoption of projects is an indicator that the users are becoming power users (based on my experience at Mural where organization and containers are proxies to deepening usage and user retention)
1–2 engineers · ~1 week including validation and rollout

Implementation Details

The usage data already exists at the server side as token consumption per conversation. The effort here is to surface this up at the front end in real time. The current usage meter in settings is updated at ~5 minute intervals. However if we are surfacing at the chat level, the API calls already provide the usage limit rates at each prompt turn.

Example below is from free plan API call:

Request Method

https://claude.ai/api/organizations/<Org id>/chat_conversations/<chat id>/completion2

Response (truncated):

event: message_limit
data: {"type":"message_limit","message_limit":{"type":"within_limit",...,"windows":{"5h":{"status":"within_limit","utilization":0.04},"7d":{"status":"within_limit","utilization":0.03}},...}}

Nudges are set to trigger when 75% of the limit is reached.

Big Bet Project Adoption · Strategic Context

Drive Project Adoption

Strategic Context

Anthropic has grown through superior products, enterprise segment focus and unwavering safety focused mission. This big bet is to move the adoption curve higher with a growth flywheel.

My strategic bet is that when the context created in claude.ai is shared with downstream automation workflows, it will generate high value outputs motivating the users to refine the context, creating a growth flywheel that continuously improves the quality of both context and the output. Projects are the context containers in Claude.ai and increasing their adoption will help users to organize, manage and hand off the context across workflows unlocking the high quality outputs and growth.

The 3-Stage Loop

Context Creation

Users build and manage the context of their work within Claude.ai. Eg. Importing memory from other AI providers, connecting external connectors, adding files into projects

Automation

The user hands off the context to downstream agentic workflow. As agents have a context based on user's domain expertise (based on Anthropic study), it delivers personalized high-value output. E.g Moving project context into Cowork, generating summaries, artifacts and markdown files

Compound

As users see the ROI of automation, they are incentivized to continuously update, expand, and refine the context of their work creating the continuous loop of higher quality outputs, refined context and product locking. E.g Updating project memory

Big Bet Project Adoption · Problem & Solution

Problem, Hypothesis & Proposed Solution

Problem

Users create valuable context across individual chats, but that context is often fragmented/silored and difficult for Claude to understand.

Main pain points are

  • Project creation is an intentional, high friction effort while the chat creation is seamless (for right reason)
  • Easier to manage multiple workstreams through existing chats than organizing into projects especially if there are numerous chats covering a single workstream.
  • Context remains siloed within the chat, leading to context rot over time.

Why Now?

Anthropic is experiencing exponential user growth from casual users with more high frequency chat sessions. The more chats that are not organized and fragmented, the chat output quality degrades leading to user churn.

Hypothesis

I believe reducing the friction in Project creation and surfacing proactive suggestions will increase adoption of active Projects because users will be more likely to organize and preserve context across projects.

I expect increased Project adoption will lead to higher-value agentic automation, deeper engagement, and stronger long-term retention as users build an increasingly valuable context layer within Claude

Success Metrics

# Proactive suggestion acceptance rate
% of WAU who have active projects
# D7 return-to-project after accepting suggestion
# WAU
# D30 retention rate
Counter: Dismissal + post-exposure opt-out rate

Kill switch: If the suggestion acceptance rate is <10% after a week of launch, pause the feature and review suggestion precision before continuing.

Proposed Solution

  • When a user logs into the product for the first time that day, the home screen lists the chats in order of "recent" as in the view all chat windows with suggested projects ( create new or add to existing) Users can accept or reject the suggestions.
  • Based on the project memory, project instruction is suggested to personalize the output. This suggestion is within the project and the user is notified through a blinking indicator in the sidebar
  • Users can also autogenerate the project memory manually.
  • Incentivize users to hand off the project memory to work with Cowork (already exists)

UX

  • Non-blocking card on home screen. Users can dismiss the matching entirely or partially.
  • Suggestions are presented as a helpful observation, not a correction.
  • Only 3 project suggestion list per day to avoid notification and suggestion fatigue
  • Opt-out: Both features are toggled as a single setting "project suggestions" and is accessible from Settings. Off by default for users.

Implementation

This is a product-layer feature and works around the current memory containment boundaries of project vs non project chats.

We use semantic match at the server-side, using project metadata. There is an existing API call to pull the projects

Request:

https://claude.ai/api/organizations/<orgid>/projects_v2?limit=40&offset=0&filter=is_creator&order_by=updated_at&is_archived=false

Big Bet Project Adoption · Risks & Rollout

Risks & Rollout

Risks and Mitigation

Risk TypeRiskMitigation
Value Risk Suggestions are wrong as incorrect projects are suggested and sensitive projects are suggested For initial release, have high precision and limit the number of suggestions until they reach the precision targets. Monitor rejection rates
Usability Risk Users see the suggestions as intrusive to their workflow Surface the background suggestions are cards in the side panel (similar to relaunch to update card) that users can view or dismiss.
Trust Risk Users views that Claude is running search on all their conversations from their past and intrusive and no control Make the project suggestion an explicit toggle that is off by default and when suggestions are surfaced get explicit user confirmation
Retention Risk User accepts the project suggestion but does not return to project to add the context Instrument D7 return to the activated project as leading indicator. If habit not formed, the project experience needs to be reviewed for more improvements before full rollout
Feasibility Risk The project memory and non project chat memory are self-contained boundaries. Updating the project meta data with the context of the chat information is sufficient to do semantic search instead of using model based search

Rollout Strategy

Phase 0 Initial rollout will be limited to internal teams to dogfood the suggestions. Kill switch: <50% of the company wide adoption
Phase 1 Rollout to 5% of users. Users have to explicitly enable it after the initial awareness. Kill Switch: <10% of test cohort adopt the feature
Phase 2 Rollout to 40% of users. Kill Switch: <10% of D7 return to project users
Phase 3 Full rollout Kill Switch: <10% increase in WAU
